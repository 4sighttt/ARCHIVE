[[AI 네이티브의 마인드셋]]
[[검증 시스템 만드는 방법 (핵심)]]
[[“프롬프트 설계 고급 구조 (실수 줄이는 방식)”]]


**핵심 요약**

- 전체 구조: **문제 정의 → 설계 → 생성 → 검증 → 결정 → 기록**
    
- 핵심은 “프롬프트”가 아니라 **작업 구조 설계**
    
- 항상 **검증 기준을 먼저 정의**
    
- 1회 실행이 아니라 **루프 반복 최적화**
    
- 결과보다 중요한 것: **재사용 가능한 프로세스**
    
- 실패 원인 90% = 문제 정의 오류 + 검증 부재
    

---

## 확장 (실전 프레임워크)

### 0. 전체 구조 (기본 루프)

```
[1] 문제 정의
→ [2] 작업 설계
→ [3] AI 생성
→ [4] 검증
→ [5] 의사결정
→ [6] 기록/템플릿화
→ 반복
```

---

## 1. 문제 정의 (Problem Framing)

가장 중요한 단계

구성 요소:

- 목표: 무엇을 얻고 싶은가
    
- 범위: 어디까지 포함/제외
    
- 제약: 시간, 데이터, 형식
    
- 성공 기준: 무엇이면 “좋다”인가
    

예:

- ❌ “시장 분석”
    
- ✅ “최근 24시간, 검증된 뉴스 기반, 시장 영향 중심 요약”
    

👉 문제 정의가 틀리면 이후 전부 무의미

---

## 2. 작업 설계 (Instruction Design)

AI에게 “일하는 방식”을 지정

포함 요소:

- 역할 (예: 분석가, 편집자)
    
- 입력 데이터 범위
    
- 출력 형식 (매우 중요)
    
- 금지 조건 (추측 금지 등)
    

구조:

```
역할:
목표:
조건:
출력 형식:
금지:
```

👉 설계 = 품질의 70%

---

## 3. AI 생성 (Execution)

여기서 중요한 건:

- 빠르게 여러 버전 생성
    
- 완벽한 1회 결과 기대 금지
    

전략:

- 1안 / 2안 / 3안 생성
    
- 서로 비교
    

👉 단일 출력 의존 = 위험

---

## 4. 검증 (Validation) — 핵심

반드시 분리된 단계로 수행

검증 유형:

1. 사실 검증 (Fact)
    
2. 논리 검증 (Logic)
    
3. 누락 검증 (Coverage)
    
4. 편향 검증 (Bias)
    

방법:

- 같은 질문을 다른 방식으로 재질문
    
- 반박 요청 (“이 분석의 오류는?”)
    

👉 검증 없는 AI 사용 = 실패 확정

---

## 5. 의사결정 (Decision)

여기서 인간이 개입

판단 기준:

- 정확성
    
- 목적 적합성
    
- 리스크
    

👉 AI는 선택하지 않는다  
→ 선택은 인간 책임

---

## 6. 기록 / 템플릿화 (Systemization)

가장 과소평가되는 단계

저장 대상:

- 잘 작동한 프롬프트
    
- 검증 기준
    
- 실패 사례
    

형태:

- 템플릿
    
- 체크리스트
    
- 규칙 문서
    

👉 이 단계부터 “생산성 폭발” 시작

---

## 핵심 압축 구조

```
문제 정의 → 설계 → 생성 → 검증 → 결정 → 기록
```

이걸 반복하면:

- 품질 ↑
    
- 속도 ↑
    
- 재사용성 ↑
    

---

## 실전 미니 템플릿 (바로 사용 가능)

```
[문제]
- 목표:
- 범위:
- 성공 기준:

[설계]
- 역할:
- 입력 조건:
- 출력 형식:
- 금지 사항:

[실행]
- 결과 3개 생성

[검증]
- 사실 오류?
- 논리 오류?
- 빠진 요소?

[결정]
- 최종 선택 + 이유

[기록]
- 잘된 점 / 수정할 점
```
