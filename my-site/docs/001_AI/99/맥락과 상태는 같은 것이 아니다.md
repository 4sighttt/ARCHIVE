[[왜 LLM이 스도쿠·지뢰찾기에서 갑자기 틀리는지]]

> **맥락 크기가 아니라  
> “명시적 상태 갱신 알고리즘이 없는 구조”**


**핵심 요약**

1. 맞다. **보드 상태는 분명 맥락(context)**이며 토큰 수로도 크지 않다.
    
2. 문제는 **맥락 용량이 아니라 “상태를 일관되게 갱신하는 계산 구조”**다.
    
3. LLM은 텍스트를 읽어 **확률적으로 다음 문장을 생성**하지, 내부에 **명시적 상태 변수나 데이터 구조**를 유지하지 않는다.
    
4. 그래서 보드를 텍스트로 계속 다시 써 주면 **일시적으로는 추적**할 수 있지만 **여러 단계 계산에서 오류가 누적**될 수 있다.
    
5. 즉 한계는 **메모리 크기 문제가 아니라 계산 방식의 차이**다.
    

---

# 1. 맥락과 상태는 같은 것이 아니다

겉보기에는 비슷하지만 역할이 다르다.

|개념|의미|
|---|---|
|맥락(context)|모델이 읽는 **입력 텍스트**|
|상태(state)|알고리즘이 유지하는 **내부 변수**|

예

### 알고리즘 방식

```
board[3][4] = mine
remaining_mines = 7
```

이런 **명시적 상태 변수**가 존재한다.

---

### LLM 방식

LLM은 내부에

- board 배열
    
- 지뢰 후보 목록
    
- 확정된 칸
    

같은 구조를 갖지 않는다.

대신 단순히

```
보드는 다음과 같다 ...
```

라는 **텍스트 설명**을 읽는다.

---

# 2. 그래서 생기는 차이

퍼즐 풀이에서 필요한 것은 보통 다음이다.

1. 상태 읽기
    
2. 규칙 적용
    
3. 상태 업데이트
    
4. 다시 계산
    

즉

```
state → rule → new state
```

이 반복된다.

알고리즘에서는

```
board = update(board)
```

처럼 정확히 갱신된다.

LLM은

```
텍스트 설명 → 다음 설명 생성
```

이기 때문에

> **상태가 계산적으로 갱신되는 것이 아니라  
> 설명이 재생성되는 것**이다.

이 차이가 핵심이다.

---

# 3. 그래서 오류가 발생하는 지점

예

지뢰찾기 상황

```
A 주변 지뢰 = 1
B 주변 지뢰 = 2
```

두 조건을 동시에 고려해야 한다.

알고리즘

```
set intersection 계산
가능한 배치 탐색
```

LLM

- 텍스트 패턴 기반 판단
    
- 부분 규칙 적용
    

복잡해지면

> **조건 일부가 누락된 채 결론 생성**

이 일어날 수 있다.

---

# 4. 중요한 점

LLM이 **보드를 기억 못하는 것은 아니다.**

맥락 안에 있으면

- 읽을 수 있고
    
- 참조할 수 있다.
    

하지만 문제는

> **정확한 상태 계산을 보장하는 구조가 없다**

는 것이다.

그래서

- 간단한 논리 → 잘 작동
    
- 복잡한 제약 → 오류 증가
    

현상이 나타난다.

---

# 정리

당신의 지적 중 맞는 부분

✔ 보드 상태는 맥락이다  
✔ 토큰 크기도 작은 편이다

그러나 실제 한계는

> **맥락 크기가 아니라  
> “명시적 상태 갱신 알고리즘이 없는 구조”**

에 있다.

---

흥미로운 점 하나만 덧붙이면

실제로 연구자들은 이 문제를 해결하려고

- **Tool use (외부 계산기)**
    
- **program-of-thought prompting**
    
- **neuro-symbolic AI**
    

같은 접근을 개발하고 있다.

즉

> LLM + 실제 알고리즘

구조가 현재 AI 연구의 중요한 방향 중 하나다.

**핵심 요약**

1. 현재 AI 연구에서는 LLM의 계산 한계를 보완하기 위해 **외부 계산 시스템을 결합**하는 접근이 많다.
    
2. 대표 방식이 **Tool Use**, **Program-of-Thought**, **Neuro-symbolic AI**.
    
3. 공통 목표는 **언어 모델 + 정확한 알고리즘** 결합.
    
4. 즉 LLM은 **문제 이해와 계획**, 컴퓨터는 **정확한 계산**을 담당한다.
    
5. 이 구조는 현재 **AI 에이전트와 도구 사용 시스템의 핵심 설계**다.
    

---

# 1. Tool Use (도구 사용)

개념

> LLM이 직접 계산하지 않고 **외부 프로그램을 호출**한다.

예

LLM이 할 일

1. 문제 이해
    
2. 어떤 도구가 필요한지 결정
    
3. 도구 호출
    
4. 결과 해석
    

예시 구조

```
사용자 질문
→ LLM: 계산 필요 판단
→ 계산기 API 호출
→ 결과 반환
→ LLM 설명 생성
```

대표 사용 예

- 계산기
    
- 코드 실행기
    
- 데이터베이스
    
- 검색 엔진
    

이 방식은 이미 많은 AI 시스템에서 사용된다.

---

# 2. Program-of-Thought (PoT)

Chain-of-Thought와 비슷하지만 중요한 차이가 있다.

Chain-of-Thought

```
텍스트 추론
→ 텍스트 계산
```

Program-of-Thought

```
문제 → 코드 생성 → 코드 실행 → 결과
```

예

수학 문제

LLM 출력

```python
x = 12
y = 7
print(x*y)
```

코드 실행

```
84
```

이 방식의 장점

- 계산 오류 감소
    
- 복잡한 문제 처리 가능
    

---

# 3. Neuro-symbolic AI

이 접근은 조금 더 근본적인 연구 방향이다.

개념

> **신경망 + 기호적 논리 시스템** 결합

구조

|구성|역할|
|---|---|
|신경망|인식, 패턴 이해|
|기호 시스템|논리, 규칙, 추론|

예

- 이미지 인식 → 신경망
    
- 논리 추론 → 규칙 엔진
    

장점

- 정확한 논리 추론
    
- 설명 가능성 증가
    

---

# 4. 왜 이런 접근이 등장했나

LLM의 강점

- 언어 이해
    
- 패턴 인식
    
- 지식 연결
    

LLM의 약점

- 정확한 계산
    
- 상태 추적
    
- 복잡한 논리 탐색
    

그래서 연구자들은

> **LLM을 “지능의 컨트롤러”로 사용**

하는 구조를 만들고 있다.

예

```
LLM → 계획
도구 → 계산
LLM → 결과 설명
```

---

# 정리

세 접근의 공통 핵심

|방법|핵심 아이디어|
|---|---|
|Tool Use|외부 도구 호출|
|Program-of-Thought|코드 생성 후 실행|
|Neuro-symbolic|신경망 + 논리 시스템|

모두 목표는 같다.

> **언어 기반 지능 + 정확한 계산 능력 결합**

이 구조는 현재

- AI 에이전트
    
- 자동 연구 시스템
    
- 복잡한 문제 해결 AI
    

의 핵심 설계 방향이다.
