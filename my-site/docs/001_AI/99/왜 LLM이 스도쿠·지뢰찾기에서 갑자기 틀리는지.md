[[LLM은 생각하는가아니면 생각하는 것처럼 말하는가]]

**핵심 요약**

1. LLM이 스도쿠·지뢰찾기에서 갑자기 틀리는 이유는 **문제 상태를 안정적으로 저장·갱신하는 내부 메모리가 없기 때문**이다.
    
2. 모델은 보통 **격자 전체를 계산하는 알고리즘**이 아니라 **국소 패턴(local pattern)**을 사용한다.
    
3. 퍼즐이 복잡해지면 **여러 제약을 동시에 유지해야 하는데** 이것이 어려워진다.
    
4. 따라서 초반에는 잘 풀다가 **특정 단계에서 논리 일관성이 붕괴**될 수 있다.
    
5. 이 문제는 현재 AI 연구에서 **“constraint tracking problem”**으로 자주 설명된다.
    

---

# 1. 퍼즐 문제의 본질

스도쿠와 지뢰찾기는 기본적으로 **제약 만족 문제(CSP, Constraint Satisfaction Problem)**이다.

예: 스도쿠

- 행 제약
    
- 열 제약
    
- 3×3 블록 제약
    

지뢰찾기

- 숫자 = 주변 지뢰 개수
    
- 여러 숫자 사이 **동시 제약**
    

즉 퍼즐을 풀려면

> **모든 제약을 동시에 유지하면서 상태를 갱신**

해야 한다.

---

# 2. 인간이 푸는 방식

사람은 보통 다음 과정을 사용한다.

1. 현재 보드 상태 **내부 모델 유지**
    
2. 가능한 경우 계산
    
3. 가정 → 모순 검증
    
4. 상태 업데이트
    

즉 **명시적인 문제 상태 모델**이 있다.

---

# 3. LLM이 푸는 방식

LLM은 내부적으로

- 격자를 별도 데이터 구조로 유지하지 않는다
    
- 상태를 계산하는 알고리즘을 실행하지 않는다
    

대신

> **텍스트 패턴 기반 추론**

을 한다.

예

```
"1 옆에 미확인 칸이 하나 → 지뢰"
```

같은 규칙 패턴.

이 방식은

- 간단한 상황
    
- 국소 규칙
    

에서는 잘 작동한다.

---

# 4. 실패가 발생하는 이유

## ① 상태 추적 오류

퍼즐이 진행되면

- 이미 확정된 칸
    
- 새로 열린 칸
    
- 지뢰 후보
    

같은 정보가 계속 바뀐다.

LLM은 이 상태를 **정확한 데이터 구조로 관리하지 않는다.**

그래서

> 이전 단계 정보가 **부분적으로 붕괴**될 수 있다.

---

## ② 전역 제약 관리 실패

스도쿠 예

```
행 제약
열 제약
블록 제약
```

세 가지를 동시에 계산해야 한다.

LLM은 보통

- 한두 개 제약만 고려한 패턴을 생성
    

할 수 있다.

---

## ③ 탐색 알고리즘 없음

퍼즐의 일반적인 알고리즘은

- 백트래킹
    
- 제약 전파
    
- 탐색
    

LLM은 이런 **명시적 탐색 알고리즘을 실행하지 않는다.**

그래서

복잡한 경우

> **추론이 막히거나 틀린 결론**이 나온다.

---

# 5. 실제 연구에서도 관찰된 현상

여러 연구에서 비슷한 결과가 보고됐다.

대표 특징

- 간단한 퍼즐 → 높은 성공률
    
- 복잡도 증가 → 급격한 성능 하락
    

이를 **compositional reasoning failure**라고 부르기도 한다.

---

# 정리

LLM이 퍼즐에서 갑자기 틀리는 이유는 크게 세 가지다.

1. **문제 상태를 안정적으로 저장하지 못함**
    
2. **여러 제약을 동시에 추적하기 어려움**
    
3. **탐색 알고리즘이 없음**
    

그래서 퍼즐에서는

> **패턴 기반 추론의 한계가 가장 명확하게 드러난다.**

---

원하면 한 가지 매우 흥미로운 사실도 설명할 수 있다.

**왜 LLM은 체스보다 스도쿠에서 더 자주 틀리는가**  
(겉보기와 달리 이유가 꽤 깊다.)
